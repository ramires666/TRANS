# План разработки переводчика технических инструкций (китайский → русский)

## 0. Референсный документ (источник требований)

В качестве образца взят реальный документ из папки проекта:
**`海康机器人千兆网口工业面阵相机用户手册V3.0.4.pdf`**
(«Руководство пользователя промышленной камеры Hikrobot с гигабитным интерфейсом», V3.0.4).

Параметры, замеренные скриптом-инспектором (PyMuPDF):

| Параметр | Значение |
|---|---|
| Страниц | 113 |
| Элементов оглавления (закладок) | 223 (4 уровня: глава/раздел/подраздел/шаг) |
| Изображений (растр) | 304 |
| Векторных drawings | сотни (графики, схемы, рамки) |
| Текстовых символов | ~70 000 |

| Производитель PDF | Microsoft® Word 2016 |

Вывод: документ — результат экспорта из MS Word, сложносоставной, с неразрывной привязкой текст «рисунок N-M», нумерованными шагами, многоуровневыми заголовками и таблицами из векторных линий. Это задаёт нижеприведенные требования.

## 1. Цель и результат

**Цель:** автоматизированно перевести техническую инструкцию (PDF) с китайского на русский язык с сохранением:
- полной структуры (заголовки 4 уровней, нумерованные списки, оглавление-закладки);
- всех изображений и векторной графики с сохранением местоположения;
- таблиц (включая «нарисованные» линиями);
- нумерации рисунков/таблиц и перекрёстных ссылок вида «如图2-1所示» / «请参考7.8 章节»;
- исходной верстки (колонки, шрифты, отступы) насколько это возможно без потери смысла.

**Результат:** PDF-файл `_RU.pdf`, визуально повторяющий оригинал, с русским текстом, заполненными метаданными и русским оглавлением-закладками.

### LLM-эндпоинт (проверено)

- Базовый URL: `http://127.0.0.1:8080/v1`
- Модель: `Qwen3.6-35B-A3B-UD-Q3_K_M.gguf` (35B параметров, Q3_K_M, n_ctx=32000)
- Режим: **reasoning** (возвращает `reasoning_content` отдельно от `content`).
- Скорость: ~153 токена/сек, ~38 токена/сек на промпт.
- Рекомендация: для чистого перевода отключать thinking флагом `chat_template_kwargs={"enable_thinking": false}` и держать `max_tokens ≥ 2048`.

## 3. Архитектура (конвейер)

```
PDF (ZH) ──► [1. Parse] ──► [2. Segment] ──► [3. Translate (LLM)] ──► [4. Rebuild] ──► PDF (RU)
                │               │                    │                      │
                ▼               ▼                    ▼                      ▼
          page/struct.json  segments.json   translations.db (sqlite)     out_ru.pdf
```

### Этап 0 — Подготовка окружения
- [ ] 0.1 Зафиксировать компакт-скрипт `_inspect.py` (есть в репо): выводит число страниц, изображений, шрифтов, TOC. Запускается на любом новом PDF.
- [ ] 0.2 Установить `openai` (или `httpx`): `pip install openai`.
- [ ] 0.3 Создать `config.yaml`: `pdf_path`, `llm_base_url`, `llm_model`, `enable_thinking=false`, `max_tokens`, `out_path`, `tmp_dir`, `glossary_path`.

### Этап 1 — Парсинг (модуль `parser.py`)
- [ ] 1.1 Открыть PDF через `fitz.open`, получить `metadata`, `get_toc()`.
- [ ] 1.2 Для каждой страницы сохранить: `page.rect`, `get_text('dict')` (блоки/линии/спаны со шрифтами и bbox), `get_images(full=True)`, `get_drawings()`, `get_text('rawdict')` для попиксельной точности.
- [ ] 1.3 Извлечь «текстовые» блоки (`type==0`) и отделить от «изображений» (`type==1`). Запомнить bbox каждого блока — это координаты для повторной заливки.
- [ ] 1.4 Детектор таблиц: гибридный — `page.find_tables()` + распознавание групп `drawings` с типом `l`/`re` (горизонтальные/вертикальные линии). На референсе `find_tables` дал 0, поэтому упор на собственный парсер «линия+текст в ячейке».
- [ ] 1.5 Сохранить промежуточный файл `intermediate/parse.json` (или `.pkl` для скорости): одна запись на блок.

### Этап 2 — Сегментация (модуль `segmenter.py`)
- [ ] 2.1 Склеить спаны в логические абзацы с учётом: заголовок (по стилю шрифта + id главы из TOC), пункт списка (Wingdings‑маркер, цифра-точка), подпись рисунка (`图N-M …`), подпись таблицы (`表N-N …`), обычный текст, ячейка таблицы.
- [ ] 2.2 Назначить каждому сегменту `id`, `type`, `section_id` (из TOC), `bbox` и `page`.
- [ ] 2.3 Выявить перекрёстные ссылки регулярками `图\d+-\d+`, `表\d+-\d+`, `第?\d+\.?\d* 章节?`, `附录[A-Z]` и пометить как «непереводимые токены-якоря».
- [ ] 2.4 Подготовить переводные единицы (≤ 1500 символов каждая — экономим окно контекста и токены): заголовок отдельно от абзаца; шаги списком целиком; ячейки таблицы поштучно.

### Этап 3 — Перевод (модуль `translator.py`)
- [ ] 3.1 Реализовать клиента к LLM с OpenAI-совместимым API (`base_url=http://127.0.0.1:8080/v1`). Использовать `openai>=1.0` с sync-клиентом.
- [ ] 3.2 Системный промпт раз(Constant): «Переводи с китайского на русский технический текст Руководства пользователя. Сохраняй термины из глоссария. НЕ переводи токены вида {{X}}, {{图N-M}}, {{Line N}}, имена параметров (Acquisition Control, Trigger Mode), маркеры списка, нумерацию. Не добавляй комментариев.»
- [ ] 3.3 Глоссарий `glossary.csv` ( zh, ru ): базовые термины (曝光→экспозиция, 触发→триггер, 帧→кадр, 增益→усиление, 快门→затвор, 表→таблица, 图→рисунок, 章节→раздел …). Накапливать по ходу.
- [ ] 3.4 Перед каждым сегментом подставлять JSON-метаданные (тип, заголовок-родитель, термины главы) в `system`/`user`-промпт — даёт контекст без большого окна.
- [ ] 3.5 Кэш в SQLite (`translations.db`): ключ = sha256(исходник + промпт + версия модели), поля `zh, ru, ts, model, accepts_review`. Дает идемпотентность и докрутку.
- [ ] 3.6 Повторы/контроль качества: post-check на наличие всех якорей (`图2-1`) в переводе; при потере — повторный запрос с замечанием «сохранить якорь».
- [ ] 3.7 Параллелизм: `ThreadPoolExecutor` (LLM-сервер однопоток-host, но batching/lmstudio поддерживает несколько запросов) — подобрать `workers` экспериментально (стart = 2).
- [ ] 3.8 Логирование: прогресс-bar (`tqdm` или встроенный), файл `log/translate.log`, JSONL-дамп неудачных сегментов для ручной правки.

### Этап 4 — Сборка (модуль `builder.py`)
- [ ] 4.1 Создать новый PDF через `fitz.open()` из копии исходника (метод «копия+правка» даёт 100% сохранение изображений и drawings): `out = fitz.open(src); out.set_metadata(...)`.
- [ ] 4.2 Для каждого текстового блока выполнить `page.add_redact_annot(bbox, text=ru, fontname="helv", fontsize=<подобрать>, align=...)` затем `page.apply_redactions(images=fitz.PDF_REDACT_IMAGE_NONE)` — максимально сохраняет картинки/вектор.
- [ ] 4.3 Подобрать кириллический шрифт через `page.insert_font(fontfile=".../DejaVuSans.ttf")` (Helvetica в PDF не знает кириллицу). Зарегистрировать один шрифт для всего документа.
- [ ] 4.4 Разместить переведённый текст в исходный bbox; при переполнении — автосжатие `fontsize` в цикле до вписывания; крайний случай — перенос на следующую страницу (отмечать warning).
- [ ] 4.5 Перевести заголовки в оглавлении-закладках: `out.set_toc([[lvl, ru_title, page] …])`, сохраняя исходную иерархию.
- [ ] 4.6 Перевести подписи рисунков/таблиц: «图2-1» → «Рис. 2-1», «表8-3» → «Табл. 8-3». Соответственно обновить перекрёстные ссылки в-body (п. 2.3 даёт карту замены).
- [ ] 4.7 Заполнить `metadata`: `title`, `author`, `subject` на русском.
- [ ] 4.8 Сохранить `out_ru.pdf`, `out` только после `out.save(..., garbage=4, deflate=True)`.

### Этап 5 — Валидация (модуль `validator.py`)
- [ ] 5.1 Сравнить число страниц (==), число изображений (==), длину TOC (==), уровень deepest заголовка (==).
- [ ] 5.2 Проверить, что все исходные якоря `图N-M`/`表N-N` присутствуют в результате (пarsiли итоговый текст).
- [ ] 5.3 Споты-человек проверяет 5 случайных страниц: визуальное сходство, читаемость, нет наезда текста на рисунок.
- [ ] 5.4 Открыть итог в Acrobat/Sumatra — закладки работают, нет ошибок шрифта («□□□» вместо букв).

## 6. Основные риски и митигация

| Риск | Митигация |
|---|---|
| Длинные русские фразы не вписываются в ширину столбца | Цикл подбора `fontsize` (12→8 pt), перенос по словам (`fitz` WRAP) |
| Картинки/вектор затираются `apply_redactions` | Параметр `images=fitz.PDF_REDACT_IMAGE_NONE`; красный бокс строго по bbox текстового блока, не по всей странице |
| «Нарисованные» таблицы ломаются при переезде текста | Переиспользовать исходные drawings; править только содержимое ячеек; при сильном смещении — перерисовать рамку `page.draw_rect` |
| LLM теряет якоря `图2-1`/термины | Системный запрет в промпте + post-check + повторный запрос; JSON-режим ответа с отдельным полем `anchors` |
| Несоответствие шрифтов (Helvetica без кириллицы) | Зарегистрировать `DejaVuSans.ttf`/`NotoSans-Regular.ttf` один раз глобально |
| Reasoning-модель тратит токены на thinking | `chat_template_kwargs={"enable_thinking": false}` + `max_tokens ≥ 2048`, контролировать по `usage.completion_tokens` |
| Один прогон по 113стр. > 1 часа | Кэш SQLite + параллелизм + resume по `page`-индексу |

## 8. Команды запуска (целевые)

```powershell
# 0. Установить зависимости
pip install openai pymupdf tqdm pyyaml

# 1. Сводка по PDF
python src\inspect.py --in "海康…V3.0.4.pdf"

# 2. Полный прогон
python run.py --in "海康…V3.0.4.pdf" --out "_RU.pdf" --resume

# 3. Только перевод (если парс готов)
python run.py --from-stage translate --resume

# 4. Валидация
python run.py --validate "_RU.pdf"
```

## 9. Критерий приёмки

1. `_RU.pdf` открывается без ошибок шрифта во всех主流-readers.
2. Кол-во страниц, изображений, уровень глубины TOC — совпадает с оригиналом.
3. Все рисунки/таблицы расположены на тех же страницах, что и в оригинале.
4. Все заголовки (1級?4級) переведены и присутствуют в навигационной панели закладок.
5. Перекрёстные ссылки «Рис. N-M», «Табл. N-N», «раздел N.M» корректны и указывают туда же, куда в оригинале.
6. Прогон `run.py` на новом PDF-ZH (без модификации кода) завершается успехом.
